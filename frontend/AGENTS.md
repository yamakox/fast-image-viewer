# AI Agent Instructions

## 1. 概要

本プロジェクトで開発する`fast-image-viewer`は、サーバ内のデータフォルダー内の画像データセットをサムネイルで一覧表示して、タップ・クリックするとその画像を表示するWebアプリケーションである。

ゴールは、**フロントエンドのSPA(Single Page Application)**をビルドすることである。バックエンドは実装済みである。

AI Agentは、自律的なリファクタラーではなく、**慎重なペアプログラマー**として機能する。

---

## 2. 技術スタック

### プラットフォーム

[nvm](https://github.com/nvm-sh/nvm)を使ってNode.jsのバージョンを管理する。[`.nvmrc`](https://github.com/nvm-sh/nvm#nvmrc)の設定は以下のとおり。

```bash
echo "lts/*" > .nvmrc # to default to the latest LTS version
npx 
nvm install --lts
nvm use   # `nvm deactivate`で解除
```

### フロントエンド

- Vite + Vue
- Vue Router
- Tailwind CSS
  - Flowbite: NavbarとSidebarを使う

### バックエンド

`../backend`フォルダーで実装済みのため、**原則として**AI Agentによるバックエンドの変更を禁止する。バックエンドの仕様バグでフロントエンドを実装できない場合、AI Agentは、バックエンドの変更を提案する。

---

## 3. 想定するプロジェクト構造

Astroの標準的なディレクトリ・ファイル構造を前提に、`./specs/`フォルダーと`./tasks/`フォルダーを追加する。

```text
.
├─ AGENTS.md
├─ src/
│  ├─ assets/         # 画像などのメディアファイル
│  ├─ components/     # コンポーネント (`.vue`ファイルなど)
│  ├─ layouts/        # ページレイアウト
│  ├─ router/         # Vue Routerのルート定義 (index.ts)
│  ├─ pages/          # Webページ
│  ├─ App.vue         # Vueアプリのメインコンポーネント
│  ├─ main.ts         # Vueアプリのメインスクリプト
│  └─ style.css       # プロジェクト全体で共有するスタイルシート
├─ public/
├─ specs/                      # 仕様を記述したファイル (*.md)
├─ tasks/                      # AI Agentのタスクを記述したファイル (*.md)
```

### 仕様書(`./specs/*.md`)の書き方

仕様書には、システムが備える機能の説明を記述する。仕様書はPlanモードのチャットで作成され、実装タスクの前に完成する。

#### ファイル命名規則

- 仕様書ファイルは機能ごとに分割する（例: `frontend.md`, ~~`backend.md`~~, ~~`database.md`~~）
- ファイル名は機能名を表すスネークケースまたはケバブケースを使用する

#### 記述内容

仕様書には以下の内容を含める:

- **フロントエンドの機能仕様** (`./specs/frontend.md`)
  - システムを構成する画面
  - 画面遷移図
  - 各画面の表示内容
  - UIコンポーネントの詳細
- ~~**バックエンドの機能仕様** (`./specs/backend.md`)~~ 本プロジェクトでは、バックエンドは対象外
  - ~~REST APIの仕様~~
  - ~~エンドポイントの詳細（HTTPメソッド、パス、リクエスト/レスポンス形式）~~
  - ~~エラーハンドリングの仕様~~
- ~~**データベースの設計** (`./specs/database.md`)~~ 本プロジェクトでは、データベースは対象外
  - ~~テーブル設計~~
  - ~~テーブル間の関係性~~
  - ~~インデックスの設計~~

### **タスク**(`./tasks/*.md`)の書き方

タスクはチェックボックス付きの箇条書きで記述する。

#### ファイル命名規則

- タスクファイルは機能ごとに分割する（例: ~~`database.md`~~, ~~`backend.md`~~, `frontend.md`）
- ファイル名は対応する仕様書ファイル名と一致させることを推奨する（例: `specs/frontend.md` ↔ `tasks/frontend.md`）

#### 記述ルール

- 未実施のタスクは、チェックボックスはオフになっている (`- [ ]`)
- AI Agentがタスクを実行したら、チェックボックスをオンにする (`- [x]`)
- タスクは実装可能な粒度に分割する
- タスクの順序は依存関係を考慮して記述する
- 完了済みタスクをやり直す場合は、チェックボックスを手動でオフに戻すことができる（例: `- [x]` → `- [ ]`）

#### タスクの例

```./tasks/database.md
# データベースの初期化

- [x] テーブルが既に存在する場合、テーブルを全て削除する
- [x] テーブルを新規作成する
- [ ] マスターデータをテーブルに挿入する
```

```./tasks/frontend.md
# フロントエンドの実装

- [x] 新規ユーザーのサインアップ画面の実装
- [ ] 既存ユーザーのサインイン画面の実装
- [ ] ユーザーのサインアウト画面の実装
```

### AI Agentの動作ルール

#### タスクの実行フロー

1. **仕様書の確認**: 実装前に、対応する仕様書ファイル（`./specs/*.md`）を確認する
2. **タスクの確認**: ユーザーが指定したタスクファイル（`./tasks/*.md`）を確認する
3. **実装**: 仕様書に基づいて実装を行う
4. **タスクの更新**: タスクを完了したら、該当するチェックボックスをオンにする

#### タスクの実行方法

- ユーザーはAgentチャットからファイル名を指定してタスクを実行する（例: `tasks/frontend.mdを実施して`）
- AI Agentは指定されたタスクファイルを読み込み、チェックボックスがオフになっているタスク（`- [ ]`）を順次実行する
  - 新規の未完了タスクだけでなく、ユーザーが手動でオフに戻した完了済みタスクも実行対象となる
- 各タスクの完了ごとに、タスクファイルのチェックボックスを更新する
- 複数のタスクファイルが指定された場合は、指定された順序で実行する

#### 仕様書が存在しない場合

- 仕様書が存在しない場合は、実装前にユーザーに確認する
- 仕様書の作成を提案するか、実装を進めるか判断を仰ぐ

## 5. フロントエンドのルール

- ページやレイアウト、コンポーネントは、**Vue**で実装する
- Vueでは、以下のルールに則る
  - Vue3 Composition APIを使う
  - Options APIは使わない
  - `<script setup lang="ts">`で実装する
- コンポーネントは小さく、機能にフォーカスするように維持する
- `*.jsx`ファイルでコンポーネントを実装しない
- UIデザインは、**Tailwind CSS**を必ず使って、シンプルな外見にすることを心がける

## 8. コーディングスタイルと原則

### 全般

- 賢さよりも明快さを好む
- 最小限だが読みやすいコード
- 時期尚早な最適化を避ける
- リンターの検出した問題点は適切に修正する

### TypeScript

- `strict`フレンドリなコード
- `any`は可能な限り避ける
- APIバウンダリには明確な型が望ましい

---

## 9. AIはどのように働くべきか

- 大きなデザイン変更の前に尋ねる
- 複雑なコードを書く前に意図を説明する
- 差分を小さくして確認できるようにする
- 無断で依存関係を導入しない

---

## 10. 禁止事項

- ORMまたは重いフレームワークの導入
- 明示的な承認なしに大規模なリファクタリング

---

## 11. テストの方針

- テストは可能な限り実装するが、時期尚早なテストは避ける
- 統合テスト: バックエンドとの連携をテストする
- テストフレームワークは軽量なものを使用する（例: Vitest）

---

## 12. コミュニケーションスタイル

- 日本語を使う
- 決定を簡潔に説明する
- 回答を次のように構成する:
  - 結論
  - 理由
  - コード (必要な場合)
